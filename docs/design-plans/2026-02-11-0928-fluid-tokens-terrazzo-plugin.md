# Fluid Tokens Terrazzo Plugin

**Created:** 2026-02-11
**Status:** Design
**Implementation Plan Doc:** `docs/implementation-plans/2026-02-11-1009-fluid-tokens-terrazzo-plugin.md`

## Overview

**What:** Replace the standalone `generate-fluid-tokens.ts` script with a custom Terrazzo plugin that computes fluid `clamp()` values and feeds them into the CSS plugin's output.

**Why:** The current two-step build (`tz build && npx tsx generate-fluid-tokens.ts`) reads the design tokens JSON twice, produces a separate `:root` block, and lives outside Terrazzo's plugin lifecycle. Moving this into a plugin gives us a single `tz build` command, a single `:root` block, proper alias resolution for downstream tokens (e.g., Button referencing fluid spacing), and config read from Terrazzo's parsed tokens map.

**Type:** Refactor

## Design Decisions

### Architecture: Custom plugin + CSS plugin output (Option B)

**Options considered:**

1. **Option A: CSS plugin config only** (`transform()` + `variableName()`) — Simplest, but the `transform()` config function only receives individual tokens, not the full tokens map. Would require reading the JSON file separately to get viewport config. No section comments.

2. **Option B: Custom plugin (transform hook) + CSS plugin output** — Plugin gets the full tokens map, computes clamp values, overrides the CSS plugin's transforms. CSS plugin handles output, naming, and alias resolution. No section comments.

3. **Option C: Custom plugin with standalone build output** — Full control (including section comments), but produces a separate file with its own `:root` block. Doesn't integrate with the CSS plugin's alias resolution.

**Decision:** Option B. The custom plugin gets clean access to the tokens map (no double JSON read), and feeding results through the CSS plugin gives us a single `:root` block and proper `var()` alias resolution for downstream component tokens. The tradeoff (no section comments) is acceptable.

### Calculation approach: Full utopia-core functions, not just calculateClamp()

**Rationale:** Terrazzo resolves cross-mode aliases to the target token's default mode, not the matching mode. This means pair tokens like `Fluid tokens.Space size pairs.3XS—2XS` get incorrect resolved values — the Max mode resolves to 2XS's min value (8px) instead of 2XS's max value (10px). Because of this, the plugin must use `calculateSpaceScale()` and `calculateTypeScale()` (which compute correct pair/step values from the Utopia config) rather than calling `calculateClamp()` on Terrazzo's resolved Min/Max values. `calculateClamp()` is used directly only for line heights, whose Min/Max modes reference `@min`/`@max` primitives directly and resolve correctly.

### Variable naming: `variableName()` on the CSS plugin

**Rationale:** The CSS plugin's `variableName()` function controls the `localID` used in both CSS output and `var()` alias references. Custom naming logic maps fluid token IDs to the expected names (e.g., `Fluid tokens.Space size.3XS` → `--space-3xs`). Non-fluid tokens fall through to Terrazzo's default naming. This ensures downstream tokens like `Button.Vertical Padding` get `var(--space-2xs-xs)` references.

### Transform override: `enforce: "post"` + setTransform() dedup

**Rationale:** The CSS plugin transforms all tokens first (including fluid ones) with their resolved px values. Our plugin runs after (`enforce: "post"`) and calls `setTransform()` without `localID` or `mode`, which matches the existing transform by token ID and updates its value in place — preserving the CSS plugin's `localID` set by `variableName()`.

## Technical Reference

### Token ID → CSS variable mapping

| Token ID pattern                                     | CSS variable            | Example                 |
| ---------------------------------------------------- | ----------------------- | ----------------------- |
| `Fluid tokens.Space size.{label}`                    | `--space-{label}`       | `--space-3xs`           |
| `Fluid tokens.Space size pairs.{a}—{b}`              | `--space-{a}-{b}`       | `--space-3xs-2xs`       |
| `Fluid tokens.Space size pairs.Custom pairs.{a}—{b}` | `--space-{a}-{b}`       | `--space-s-l`           |
| `Fluid tokens.Font size.Step {n}`                    | `--step-{n}`            | `--step-0`, `--step--1` |
| `Fluid tokens.Line height body.Step {n}`             | `--lh-body-step-{n}`    | `--lh-body-step-0`      |
| `Fluid tokens.Line height heading.Step {n}`          | `--lh-heading-step-{n}` | `--lh-heading-step-0`   |

All labels are lowercased. The em dash (—) in pair names becomes a hyphen (-).

### Terrazzo plugin API usage

- **Hook:** `transform` with `enforce: "post"`
- **Input:** `tokens` map (read viewport/space/type config), `setTransform` (override CSS plugin's transforms)
- **Format:** `"css"` (shared with the CSS plugin via the formats registry)
- **Dependencies:** `utopia-core` (`calculateSpaceScale`, `calculateTypeScale`, `calculateClamp`)

### Known limitation: Terrazzo cross-mode alias resolution

Terrazzo resolves token aliases to the target's default mode regardless of the referencing mode's context. This affects pair tokens whose modes reference tokens that themselves have modes. The plugin works around this by computing values from the Utopia config directly rather than relying on Terrazzo's resolved mode values.
