# Storybook Token Display

Date: 2026-02-11

## Context

We decided to drop the `terrazzo-plugin-structure.mjs` post-processing script that organized and commented `tokens.css`. The structured/commented output was partly motivated by the `storybook-design-token` addon, which requires CSS comment annotations to categorize tokens for display.

With that approach removed, we need a different strategy for displaying design tokens in Storybook so designers can verify the token pipeline output.

## The constraint

`tokens.css` is autogenerated by Terrazzo. Any approach that requires annotating or structuring that file is off the table.

## Two data sources available

1. **`design.tokens.json`** — Figma export (DTCG format). Has rich structure: groups, types, min/max mode values. This is the semantic source of truth.
2. **`tokens.css`** — Flat CSS custom properties with final `clamp()` values. The runtime artifact.

## Approaches evaluated

### 1. `@etchteam/storybook-addon-design-token-tables` (v2.1.0, Feb 2026)

The lowest-friction addon option. Reads raw CSS via Vite's `?raw` import — no annotations needed. You configure prefix-to-display-type mappings:

```ts
// preview.ts
import rawTokens from '../src/tokens/tokens.css?raw';

parameters: {
  designTokenTables: {
    tokens: [rawTokens],
    collections: {
      'color-primitives': 'color',
      'space': 'spacing',
      'step': 'fontSize',
    },
  },
}
```

Then use `<Tokens collection="color-primitives" />` in MDX pages. Our naming convention (`--color-primitives-*`, `--space-*`, `--step-*`, `--lh-body-*`) maps cleanly to its prefix-based matching. Built-in display types for color, spacing, fontSize, etc.

**Downside:** Fluid `clamp()` values show as raw strings, not live-rendered. It's a table of text, not a visual demo.

### 2. Custom React components as stories

Build small token viewer components that actually _use_ the CSS variables. A spacing viewer renders divs sized with `var(--space-m)` so it visually resizes with the viewport. A type viewer renders actual text at each step. A color viewer renders swatches using the var values.

For data, you can either:

- Import `design.tokens.json` for the token names, grouping, and descriptions
- Use `getComputedStyle()` at runtime to show the resolved pixel value at the current viewport width
- Use the CSS variable directly in `style` props so the element demonstrates the actual fluid behavior

This is the only approach that can properly demonstrate fluid tokens. The designer resizes the Storybook iframe and watches spacing/type scale respond.

**Downside:** More upfront build time. But once built, it reads from existing sources and auto-updates when tokens change.

### 3. Storybook's built-in doc blocks (`ColorPalette`, `Typeset`)

Already installed via `@storybook/addon-docs`. `ColorPalette`/`ColorItem` is decent for color swatches. `Typeset` is limited — takes pixel numbers, can't handle `clamp()` values.

Could work as a quick solution for colors specifically. Not viable for fluid spacing/type without custom components.

### 4. `storybook-design-token` addon (v5.0.0)

The well-known option. Has 13 built-in presenters (Color, Spacing, FontSize, LineHeight, etc.) and looks polished. But it requires `@tokens`/`@presenter` comment annotations in the CSS file. Since `tokens.css` is autogenerated, you'd have to inject annotations during the build — exactly the kind of post-processing we decided to stop doing.

## Comparison

| Criteria                      | etchteam addon        | Custom components    | Built-in doc blocks    | storybook-design-token |
| ----------------------------- | --------------------- | -------------------- | ---------------------- | ---------------------- |
| Requires modifying tokens.css | No                    | No                   | No                     | Yes (annotations)      |
| Handles fluid clamp() values  | Shows raw string      | Live visual demo     | No (needs px)          | Shows raw string       |
| Presentation control          | Medium (table format) | Maximum (full React) | Low (fixed styles)     | Medium (13 presenters) |
| Initial setup effort          | Low                   | High                 | Low                    | Medium                 |
| Ongoing maintenance           | Low                   | Low                  | Low-Medium             | High                   |
| New dependency                | Yes                   | No                   | No (already installed) | Yes                    |

## Recommendation

**Build ~5 custom React viewer components:** colors, space sizes, space pairs, font sizes, line heights. Each reads grouping info from `design.tokens.json` and renders elements using the actual CSS variables, so fluid tokens visually respond to viewport resizing. No addon dependencies, no annotation requirements, auto-updates when tokens change.

**Fallback:** If custom components feel like too much upfront, `@etchteam/storybook-addon-design-token-tables` can cover colors and font families/weights with minimal setup. But it only shows raw `clamp()` strings for fluid tokens — you'd still need custom components for spacing and type.
